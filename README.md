# Smart Document Processing System

End-to-end system that ingests invoices and purchase orders from multiple file formats, extracts structured fields with an LLM, runs deterministic validation, and exposes a review UI for manual correction.

**Live demo:** _your-vercel-url-here_
**Backend API docs:** _your-render-url_/docs

---

## Tech stack

**Backend** — Python 3.12 + FastAPI + SQLite + SQLAlchemy
**LLM** — Google Gemini 2.0 Flash via the `google-genai` SDK (free tier)
**OCR** — Tesseract (via `pytesseract`) for image inputs
**PDF** — `pdfplumber` for text-based PDFs
**Frontend** — Angular 17+ standalone components + Angular Material
**Deployment** — Render (backend), Vercel (frontend)

---

## How it works

```
File upload
    ↓
Per-format parser (PDF / Image / CSV / TXT)  →  raw text + structured hints (CSV only)
    ↓
Gemini extraction with JSON-mode + schema  →  ExtractedData (Pydantic)
    ↓
Validation engine (totals, dates, math, duplicates, missing fields)  →  list of issues
    ↓
SQLite persistence  +  status (Uploaded / Needs Review / Validated / Rejected)
    ↓
Angular review UI for manual corrections
```

The split between deterministic parsers and the LLM is intentional. Structured formats like CSV give us clean line items without guessing, while the LLM only handles the unstructured parts. For PDF, image, and TXT inputs (where structure is irregular), the LLM does the bulk of extraction with a strict "never invent data — return null" instruction.

## Local setup

### Prerequisites
- Python 3.12 (3.13/3.14 will not work because Pillow/pydantic-core have no prebuilt wheels yet)
- Node.js 18+
- Tesseract OCR (only required to process image uploads)
  - macOS: `brew install tesseract`
  - Ubuntu: `sudo apt install tesseract-ocr`
  - Windows: https://github.com/UB-Mannheim/tesseract/wiki
- A Gemini API key (free at https://aistudio.google.com/app/apikey)

### Backend

```bash
cd backend
python -m venv venv
# Windows:
venv\Scripts\activate
# macOS/Linux:
source venv/bin/activate

pip install -r requirements.txt
cp .env.example .env
# Edit .env: set GEMINI_API_KEY (and TESSERACT_CMD on Windows if not on PATH)

uvicorn app.main:app --reload
# Open http://localhost:8000/docs for the API explorer
```

### Frontend

```bash
cd frontend
npm install
ng serve
# Open http://localhost:4200
```

### Tests

```bash
cd backend
pytest -v
```

## API overview

`POST /api/documents` — upload a file, returns extracted data and validation issues
`GET /api/documents` — list all documents (optional `?status_filter=...`)
`GET /api/documents/stats` — counts by status and validated totals by currency
`GET /api/documents/{id}` — full document detail
`PUT /api/documents/{id}` — save manual corrections (re-validates)
`PUT /api/documents/{id}/status` — change workflow status
`DELETE /api/documents/{id}`

Full interactive documentation: `/docs` (Swagger UI generated by FastAPI).

## Validation logic

Implemented in `backend/app/services/validator.py` and covered by unit tests in `backend/tests/test_validator.py`:

1. **Document type recognised** — error if `unknown` (e.g. random screenshot mistakenly uploaded).
2. **Required fields present** — supplier, document number, total are errors; issue date and currency are warnings.
3. **Date sanity** — issue date and due date must parse as ISO dates; due date cannot be before issue date; future issue dates flagged as warnings.
4. **Line item math** — for each line where quantity, unit price and total are present, verifies `total ≈ quantity × unit price`.
5. **Total consistency** — verifies `subtotal + tax ≈ total` and `sum(line_items.total) ≈ subtotal`.
6. **Duplicate detection** — same supplier + document number combination existing in another row is flagged as an error.

Money comparisons use a tolerance of max(0.02, 0.5% of expected) to handle rounding without false positives.

## AI usage

I used Claude (the Anthropic assistant) to scaffold the project structure, write boilerplate (DB models, API plumbing, Material UI components), and pair-program the Gemini prompt iterations. Architectural decisions, the validation logic, the parser fallback chain, and the file structure were my own — I treated the assistant as a faster way to type code I understood, not as a black-box solution generator.

For document extraction itself, I use Google Gemini 2.0 Flash with structured JSON output. The system prompt explicitly forbids invented data: missing fields must come back as `null` rather than guessed strings. Pydantic validates the response and the extractor salvages partial responses if Gemini returns slightly off-schema data.

## Known limitations / improvements I would make

- **Scanned PDFs**: only text-based PDFs are supported. The fallback (rasterize each page with PyMuPDF, run OCR) was scoped out for time but is a straightforward add.
- **OCR preprocessing**: image OCR uses simple grayscale conversion. Deskew + adaptive thresholding via OpenCV would improve accuracy on real-world photos.
- **Auth**: this is single-user. Adding JWT auth and per-user document isolation is the obvious next step.
- **Postgres in prod**: SQLite is fine for the demo but does not survive container restarts on Render free tier. Switching to a managed Postgres (e.g. Supabase free tier) is one env-var change.
- **Background processing**: extraction is currently synchronous in the upload request, so large files block. Moving Gemini calls to a Celery/RQ worker would let the UI show a progress state.
- **More validation rules**: e.g. cross-check supplier name against a known suppliers list, sanity-check tax rates per jurisdiction.
- **CSV header metadata**: some CSV uploads might have free-text header rows (supplier name, doc number) above the table. The current parser only extracts line items; the LLM is called on the raw CSV text to catch this, but a dedicated pre-pass would be more robust.
